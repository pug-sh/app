# India Point-of-View Map Rendering

Both maps in the dashboard render national boundaries per India's official point of view:
Jammu & Kashmir — including Pakistan-administered Kashmir (Azad Kashmir + Gilgit-Baltistan),
the Shaksgam valley, and Aksai Chin — is shown as part of India. This matches how
[openstreetmap.in](https://www.openstreetmap.in/) corrects the same OpenStreetMap data, and is a
legal requirement for maps published for the Indian market.

Upstream map data ships the *de-facto* worldview instead (PoK drawn as Pakistan, Aksai Chin as
China, the Line of Control drawn as the international boundary), so both maps apply a
precomputed correction at render time. No external tile service is involved — everything stays
self-hosted.

## The two maps

| Map | Component | Base data | Correction |
| --- | --- | --- | --- |
| Live visitors map (`/live`) | `src/components/live-map/visitor-map.tsx` | OSM-derived Protomaps PMTiles | Style-level: hide disputed lines, overlay POV lines |
| Activity choropleth (overview + dashboards) | `src/components/activity-heatmap-map.tsx` | `world-atlas` countries-110m (Natural Earth) | Geometry swap for India/Pakistan/China |

Both corrections consume JSON committed under `src/lib/data/`, generated by
`scripts/build-india-pov-data.ts` (see "Regenerating" below).

## Choropleth: geometry patch

`src/lib/data/india-pov-countries-patch.json` holds replacement MultiPolygons for India (M49
`356`), Pakistan (`586`), and China (`156`). Inside a Kashmir-sector bounding box the world-atlas
110m shapes are swapped for Natural Earth's official India-POV outlines
(`ne_10m_admin_0_countries_ind`, simplified); outside the box the original 110m geometry is kept
so shared borders with all other neighbours stay untouched.

`src/lib/world-countries.ts` substitutes these geometries when building `WORLD_COUNTRIES`, so the
fill, the outline, and hover hit-testing all follow the POV shapes — hovering Gilgit or Aksai
Chin reports India.

## Live map: boundary style transforms

The Protomaps tiles flag boundary segments OSM marks as disputed with `disputed: true`.
`src/lib/live-map/basemap.ts` applies two transforms on top of the stock protomaps theme:

1. **Hide every disputed segment** — both boundary layers get an extra
   `['!=', 'disputed', true]` filter. This removes the Line of Control and the Line of Actual
   Control, but also removes lines that must stay (see below), which is why step 2 exists.
2. **Overlay the POV line set** — `src/lib/data/india-pov-boundary-lines.json` is added as a
   GeoJSON source, drawn with the exact paint of the theme's `boundaries_country` layer (cloned
   per light/dark theme), slotted directly above it.

The overlay contains two kinds of lines:

- **Claim lines with no de-facto boundary at all**, from Natural Earth's India-POV dataset:
  the western J&K boundary (separating PoK from Pakistan proper), the northern Shaksgam
  frontier, and the eastern edge of Aksai Chin.
- **Disputed-flagged OSM segments that official Indian maps draw as plain borders**, re-extracted
  verbatim from the tiles: the Jammu "working boundary", the middle sector (Himachal/Uttarakhand
  and Kalapani near Nepal), the McMahon line along Arunachal Pradesh, and every disputed line
  outside the India region (Crimea, Western Sahara, ...), so the rest of the world renders
  exactly as the stock theme would. Reusing the tiles' own geometry makes these reconnect
  seamlessly with the still-visible non-disputed lines.

Net effect — hidden: Line of Control, LAC around Aksai Chin, the 1963 Pakistan–China boundary.
Drawn: everything else, plus India's claimed outer boundary.

### How segments are classified

For each disputed line extracted from the tiles, a segment is kept (redrawn) when it lies outside
the India zone (lon 60–100, lat 5–40), or when it coincides (within ~4 km) with a boundary India
recognises: the Natural Earth India-POV outline, or the de-facto ring of any country **except**
India, Pakistan, and China. Those three rings are exactly the ones that carry the LoC, the LAC,
and the 1963 line — so those stay hidden — while e.g. the Nepal, Bhutan, and Afghanistan rings
keep their disputed stretches visible.

## Build pipeline

Two scripts feed the maps. Neither runs as part of `bun run build` — the app build only bundles
their committed/fetched outputs.

```
scripts/fetch-map-assets.ts          scripts/build-india-pov-data.ts
        │                                       │
        ▼                                       ▼
public/basemap.pmtiles  ──reads──►  src/lib/data/india-pov-countries-patch.json
public/fonts/                       src/lib/data/india-pov-boundary-lines.json
(gitignored, self-hosted)           (committed, bundled into the JS)
```

### 1. Map assets — `bun run fetch:map-assets`

Populates `public/` with the self-hosted MapLibre assets (both gitignored):

- `public/basemap.pmtiles` — a z0–z8 global extract of the Protomaps daily planet build, pinned
  via `BUILD_DATE` in `scripts/fetch-map-assets.ts` (override with `PROTOMAPS_BUILD_DATE=YYYYMMDD`).
  Requires the `pmtiles` CLI (go-pmtiles); the extract uses HTTP range requests, so only the
  low-zoom tiles (~500 MB) are downloaded, not the full planet.
- `public/fonts/` — Noto Sans glyph PBFs required by the theme's labels.

Local dev serves both from `public/`. Production serves them from an object store (Cloudflare
R2) via `VITE_MAP_ASSETS_URL` because the basemap is too large for our static host — after
re-fetching, upload the new files to that bucket. The bucket must allow CORS from the app origin
and honour Range requests.

To re-fetch, delete `public/basemap.pmtiles` and/or `public/fonts/` and run the script again.

### 2. POV data — `bun run generate:pov-data`

Regenerates the two committed JSONs in `src/lib/data/`. Run it **after** `fetch:map-assets`
(it decodes disputed lines out of `public/basemap.pmtiles`).

Inputs:

- `ne_10m_admin_0_countries` + `ne_10m_admin_0_countries_ind` — Natural Earth 10m de-facto and
  India-POV worldviews, fetched from the `nvkelso/natural-earth-vector` GitHub repo and cached
  in `/tmp/ne` (delete the cache to force a re-download).
- `node_modules/world-atlas/countries-110m.json` — the choropleth base.
- `public/basemap.pmtiles` — the live-map tiles.

Stages:

1. **Choropleth patch** — for India/Pakistan/China: cut the Kashmir bbox out of the 110m shape,
   insert the simplified (Douglas-Peucker, 0.02°) 10m POV geometry for that window, union the
   two, round to 4 decimals. → `india-pov-countries-patch.json` (~14 KB).
2. **Claim lines** — walk the POV India ring and keep runs of segments in the Kashmir bbox that
   appear on no de-facto country ring (exact segment-key diff at 6 decimals). These are the lines
   OSM has nothing for.
3. **Disputed-line extraction** — sweep every tile at z5 to find disputed country-level
   boundaries, re-extract those areas at z7 (fine enough for the basemap's z8 maximum), then
   classify each segment (see "How segments are classified" above). Kept lines are simplified at
   0.005°.
4. Stages 2 + 3 merge into one MultiLineString → `india-pov-boundary-lines.json` (~65 KB).

The console output prints sanity numbers — expect roughly: India area +4%, Pakistan −10%,
China −0.3%, three claim-line segments, and ~120 raw / ~140 kept disputed lines. Large deviations
mean an upstream data change worth inspecting.

When to re-run: bumping the pinned Protomaps build date, bumping `world-atlas`, or a Natural
Earth data refresh. Commit the regenerated JSONs.

### 3. Verifying after a regeneration

There is no automated test; eyeball the result:

- Quick offline check: render the patch + overlay to SVG (the script's outputs are plain
  GeoJSON; any scratch renderer works) and confirm the J&K outline matches the official
  depiction with no line cutting through the middle of Kashmir.
- In-app: `bun run dev`, then check the live map (`/live`) around Jammu (the working boundary
  must connect the international border to the claim line — no gap), Gilgit-Baltistan, Aksai
  Chin, and Arunachal Pradesh (McMahon line present); and the overview activity map for the
  India fill extending over all of J&K. Check light and dark themes.

`polygon-clipping` is a devDependency used only by this script. The generated JSONs are excluded
from Biome formatting via `biome.json`.

## Known trade-offs

- Disputed **region-level** (state/province) lines are hidden globally, not just in Kashmir.
  Inside J&K that is intentional (no internal LoC-shaped state line); elsewhere the loss is a
  handful of faint admin-1 lines.
- OSM **place labels** are untouched: "Azad Kashmir" / "Gilgit-Baltistan" still appear as region
  labels at higher zooms. Boundaries, not labels, are what the depiction rules govern.
- The western J&K claim line (Natural Earth, ~10m scale) can sit a few km from the OSM
  admin-4 region lines visible underneath at the basemap's top zoom (z8).
- The choropleth patch stitches 10m POV geometry into 110m shapes at the Kashmir bbox edge; the
  resolution seam is sub-pixel at the sizes the choropleth renders.
